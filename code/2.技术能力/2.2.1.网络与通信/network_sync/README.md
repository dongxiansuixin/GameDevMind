# 网游网络同步方案对比演示

纯本地模拟，无网络依赖。通过两个独立的 C++ 程序直观展示**帧同步 (Lockstep)** 与**状态同步 (State Sync)** 的核心原理与差异。

---

## 快速运行

### 方式一：CMake 构建
```bash
cd network_sync
mkdir -p build && cd build
cmake ..
make
./lockstep      # 帧同步演示
./state_sync    # 状态同步演示
```

### 方式二：直接编译
```bash
cd network_sync
g++ -std=c++17 -O2 lockstep.cpp -o lockstep && ./lockstep
g++ -std=c++17 -O2 state_sync.cpp -o state_sync && ./state_sync
```

## 对比总览

| 维度 | 帧同步 (Lockstep) | 状态同步 (State Sync) |
|---|---|---|
| **核心思想** | 同步输入，各自计算 | 服务器计算，下发状态 |
| **数据流向** | 客户端→服务器(输入)→广播所有客户端 | 客户端→服务器(输入)→服务器→客户端(状态) |
| **确定性要求** | ⭐⭐⭐ 极高。逻辑必须完全确定性（禁止随机/浮点/系统时间） | ⭐ 低。服务器是权威，客户端差异会被校正 |
| **适用游戏类型** | RTS（星际争霸）、MOBA（DOTA2）、回合制策略 | FPS（CS:GO/Valorant）、MMO（魔兽世界）、格斗游戏 |
| **网络带宽** | 低 — 仅传输输入指令（每帧几十字节） | 较高 — 传输实体状态（位置/血量/动画等） |
| **网络延迟要求** | ⭐⭐⭐ 极高。一帧延迟阻塞所有玩家（常需延迟补偿） | ⭐⭐ 中等。客户端预测掩盖延迟，但高延迟导致明显回弹 |
| **玩家数量上限** | 高 — 带宽随玩家增长慢（只传输入） | 中低 — 带宽随实体数量线性增长 |
| **断线重连** | 困难 — 需从头重放全部帧序列 | 容易 — 直接拉取最新服务器状态 |
| **观战/录像** | 极其轻量 — 仅存储输入序列即可完整回放 | 需额外实现 — 状态快照或回放系统 |
| **作弊防护** | ⭐ 弱。客户端拥有全部游戏状态，易开发地图全开等外挂 | ⭐⭐⭐ 强。服务器校验一切，客户端仅提交意图 |
| **实现复杂度** | 中 — 确定性逻辑 + 同步调试困难 | 高 — 预测/校正/插值/延迟补偿体系复杂 |
| **典型引擎/案例** | 星际争霸2、帝国时代、DOTA2、王者荣耀 | Unreal Engine(Replication)、Unity Netcode、CS:GO、Valorant |
| **演示文件** | `lockstep.cpp` | `state_sync.cpp` |

## 原理说明

### 1. 帧同步 (Lockstep)

```
时间轴 →

客户端A: [输入]──→[等待所有输入]──→[执行帧N]──→[输入]──→...
                ↘                              ↗
服务器:          [收集广播输入]
                ↗                              ↘
客户端B: [输入]──→[等待所有输入]──→[执行帧N]──→[输入]──→...
```

**核心流程：**

1. **输入收集阶段** — 每个客户端将自己的输入发送给服务器（或P2P广播）
2. **等待同步** — 所有客户端等待收到所有玩家的该帧输入
3. **确定性执行** — 所有客户端使用完全相同的输入、在完全相同的初始状态下，执行完全相同的游戏逻辑
4. **结果一致** — 由于逻辑的确定性，所有客户端产生完全相同的新状态

**关键约束：**
- 游戏逻辑**必须确定性**：禁止 `rand()`、系统时间、浮点数非确定性行为
- **固定帧率**：所有客户端必须以相同 Tick Rate 推进（如 30 tick/s）
- **阻塞式同步**：任一玩家延迟 → 所有玩家等待（常通过延迟补偿/缓冲缓解）

**优点：** 极致带宽节省、完美录像/回放、适合大量单位  
**缺点：** 延迟敏感、确定性调试困难、易作弊

---

### 2. 状态同步 (State Sync)

```
时间轴 →

客户端:  [输入→预测移动]────[预测移动]────[收到Server状态→⚡回弹校正]
              │                                      ↑
服务器:  [收输入]────[权威计算]────[下发状态]────────┘
           ↑ 延迟 ↓
```

**核心流程：**

1. **客户端输入** — 玩家按下按键
2. **客户端预测 (Client Prediction)** — 客户端立即本地移动，不等待服务器确认（消除感知延迟）
3. **发送命令** — 输入发送给服务器
4. **服务器权威计算** — 服务器按照自身规则处理移动（可校验合法性）
5. **回传状态** — 服务器将权威状态发给客户端（有网络延迟）
6. **伺服校正 (Reconciliation)** — 客户端收到服务器状态后，对比本地预测：
   - 若一致 → 无需处理
   - 若不一致 → **回弹 (Rubber-banding)** 到服务器位置，重放未确认的预测

**关键机制：**
- 服务器是唯一**真相源 (Source of Truth)**
- 客户端做**乐观更新**提升手感
- 偏差时**服务器校正**保证最终一致性

**优点：** 反作弊强、断线重连简单、适合低玩家数高实时性场景  
**缺点：** 带宽较高、回弹体验差、实现复杂

## 演示文件说明

### `lockstep.cpp` — 帧同步模拟

```
场景: 10×10 网格，2个玩家回合制移动
机制: 两个客户端接收相同输入序列 → 独立计算 → 并排展示状态 → 验证一致性
输出: 每帧并排显示两个客户端的网格 + 状态哈希对比 + 最终一致性判定
```

**演示的核心概念：**
- ✅ 确定性逻辑：相同输入 → 相同输出
- ✅ 输入驱动：客户端收集所有玩家输入后才推进
- ✅ 固定帧率：所有客户端以相同节奏执行
- ✅ 无服务器权威：客户端自行计算游戏状态

### `state_sync.cpp` — 状态同步模拟

```
场景: 玩家持续向右移动
机制: 客户端预测速度 1.0 → 服务器权威速度 0.7 → 延迟 3 帧
输出: 每帧显示预测位置 vs 权威位置 + 回弹事件记录
```

**演示的核心概念：**
- ✅ 服务器权威：服务器状态是唯一真相
- ✅ 客户端预测：输入后立即本地移动
- ✅ 伺服校正：服务器状态到达后修正偏差 → "回弹"
- ✅ 延迟影响：延迟越高，偏差积累越大，回弹越剧烈

## 修改参数体验不同效果

在代码中可直接修改以下常量来观察行为变化：

### lockstep.cpp
- `GRID_W` / `GRID_H` — 网格大小
- `TOTAL_FRAMES` — 模拟帧数
- `input_sequence` — 自定义输入序列

### state_sync.cpp
- `CLIENT_SPEED` — 客户端预测速度 (调大 → 偏差更大)
- `SERVER_SPEED` — 服务器权威速度 (调小 → 偏差更大)
- `LATENCY_FRAMES` — 模拟延迟帧数 (调大 → 回弹更频繁)
- `TOTAL_FRAMES` — 模拟总帧数
