## 代码质量管理
<p>
从长期来说，高质量、设计合理的代码可以给项目带来高效性、稳定性和产品的优良体验。代码质量管理包括编码规范、代码重构、代码审查等方面，是保证项目长期可维护性的重要手段。
</p>

*Code Review,Code Refactoring,Code Style,编码规范,代码重构,单元测试,AI Coding*

## 编码规范

| 维度 | 内容 |
|------|------|
| **作用** | 定义代码编写的一致性和标准，确保团队成员编写的代码风格统一 |
| **应用场景** | 所有代码编写场景<br>多人协作的项目<br>代码审查<br>新成员入职培训 |
| **做什么的？** | 定义代码编写的一致性和标准，确保团队成员编写的代码风格统一 |
| **在哪用？** | 所有代码编写场景，特别是多人协作的项目 |

| 问题 | 解决方向 |
|------|----------|
| **如何制定适合团队的编码规范？** | 提高代码质量：通过统一的规范减少常见错误和潜在问题<br>参考行业标准（如Google风格指南）<br>根据团队实际情况调整<br>团队讨论达成共识<br>使用自动化工具检查<br>定期更新规范<br>候选规范：[谷歌开源项目风格指南](https://zh-google-styleguide.readthedocs.io/en/latest/google-cpp-styleguide/) — 可以参考并适配到自己的项目 |
| **如何确保团队成员遵守编码规范？** | 提高代码可读性（提高团队效率）：统一的风格让团队成员更容易理解和维护代码<br>使用代码格式化工具（如Prettier、clang-format）<br>集成到CI/CD流程<br>代码审查时检查<br>提供规范和工具文档<br>新成员入职培训 |
| **如何平衡规范性和灵活性？** | 核心规范必须遵守<br>允许团队自定义部分规范<br>定期审查和调整<br>避免过度规范 |
| **编码规范的具体内容** | 举例：<br>命名规范：变量、函数、类等的命名约定，如驼峰命名、下划线命名等<br>注释：注释的格式、内容和位置规范<br>标识符：全局变量的使用规范，避免过度使用全局变量<br>代码结构：switch都带上break，避免fall-through导致的bug |

| 要点和思考方向 |
|----------------|
| 编码规范是代码质量管理的基础 |
| 通过统一的规范减少常见错误和潜在问题 |
| 统一的风格让团队成员更容易理解和维护代码 |
| 需要平衡规范性和灵活性 |
| 使用自动化工具确保规范执行 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「编码规范」「命名」「注释」「Prettier」「clang-format」「ESLint」 |
| **方法** | 让 AI 制定规范、生成检查规则、设计 CI 集成 |
| **应用** | 规范制定、格式化配置、规范检查脚本 |
| **提示词范例** | 「团队需 C#/C++ 编码规范，请参考 Google 风格指南制定命名、注释、switch-break 等规范，并给出 Prettier/clang-format 的配置要点及 CI 集成方式」 |

## 代码重构

| 维度 | 内容 |
|------|------|
| **作用** | 在不改变代码外部行为的前提下，改善代码的内部结构，提高代码质量 |
| **应用场景** | 当代码变得难以维护、理解或扩展时<br>消除技术债务<br>提高代码可读性和可维护性<br>优化代码性能<br>准备添加新功能前的代码整理 |
| **做什么的？** | 在不改变代码外部行为的前提下，改善代码的内部结构，提高代码质量 |
| **在哪用？** | 当代码变得难以维护、理解或扩展时，需要进行重构 |

| 问题 | 解决方向 |
|------|----------|
| **如何判断何时需要重构？** | 代码有"坏味道"（Code Smell）：长函数、重复代码、过大的类等<br>添加新功能困难，需要修改多处代码<br>代码难以理解，需要大量注释才能看懂<br>测试难以编写，依赖关系复杂<br>性能问题，需要优化代码结构 |
| **如何安全地进行重构？** | 建议：要有比较好的测试机制支撑，比如单元测试，自动测试等。否则重构的成本会很大 — 测试是重构的安全网，没有测试的重构风险很高<br>建立完善的测试机制（单元测试、集成测试）<br>小步重构，每次只做小的改动<br>使用版本控制，便于回滚<br>重构前先理解代码，不要盲目重构<br>使用重构工具（IDE的重构功能） |
| **如何避免重构引入新的bug？** | 重构前确保有足够的测试覆盖<br>重构后运行所有测试<br>使用自动化测试验证行为不变<br>代码审查，让其他人检查重构<br>逐步重构，每次验证后再继续 |
| **重构的具体方法** | 让代码变得更清晰、更稳定、更可维护：通过重构消除技术债务，提高代码质量<br>方法举例：<br> - 提炼过长的函数：将长函数拆分成多个小函数，提高可读性<br> - 降低耦合：减少模块之间的依赖关系，提高代码的模块化程度<br> - 将方法放到合适的类中去：遵循单一职责原则，将方法放在最合适的类中 |
| **单元测试支持** | 单元测试：<br>依赖：持续集成 — 将单元测试集成到CI/CD流程中，自动运行测试<br>断言：使用断言验证代码行为是否符合预期<br>Arrange, Act, Assert (AAA Pattern)：测试的标准结构模式 — Arrange准备测试数据和环境，Act执行被测试的代码，Assert验证结果是否符合预期 |

| 要点和思考方向 |
|----------------|
| 代码重构是提高代码质量的重要手段 |
| 通过重构消除技术债务，让代码更清晰、更稳定、更可维护 |
| 测试是重构的安全网，没有测试的重构风险很高 |
| 小步重构，逐步改善，避免大范围改动 |
| 使用重构工具和自动化测试提高重构效率 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「重构」「提炼函数」「降低耦合」「单元测试」「AAA」 |
| **方法** | 让 AI 识别坏味道、设计重构步骤、生成单元测试 |
| **应用** | 重构方案、测试用例、小步重构 |
| **提示词范例** | 「这段 C# 函数过长、重复多，请按提炼函数、降低耦合的思路设计重构步骤，并给出 AAA 模式的单元测试用例以保障行为不变」 |

## 代码审查（Code Review）

| 维度 | 内容 |
|------|------|
| **作用** | 通过团队成员对代码的审查，发现潜在问题，分享知识，提高代码质量 |
| **应用场景** | 所有代码提交前<br>重要功能或核心代码的修改<br>新成员代码的审查<br>代码重构后的审查 |
| **做什么的？** | 通过团队成员对代码的审查，发现潜在问题，分享知识，提高代码质量 |
| **在哪用？** | 所有代码提交前，特别是重要功能或核心代码的修改 |

| 问题 | 解决方向 |
|------|----------|
| **如何高效地进行代码审查？** | 在审查过程中共享知识：通过代码审查，团队成员可以学习不同的编程技巧和最佳实践<br>建立代码审查流程和规范<br>使用代码审查工具提高效率<br>明确审查重点和检查清单<br>控制审查代码量，避免过大<br>及时反馈，避免阻塞 |
| **如何平衡审查的严格程度和开发效率？** | 保证开发质量：在代码合并前发现问题，避免bug进入主分支<br>根据代码重要性调整审查严格程度<br>核心代码严格审查，工具代码可以放宽<br>建立审查优先级机制<br>使用自动化工具辅助审查<br>避免过度审查影响开发效率 |
| **如何让代码审查成为学习机会而非负担？** | 获得不同角度的思考：不同的人有不同的视角，可以发现作者忽略的问题<br>避免只有一个人熟悉系统：通过代码审查，让更多人了解系统的实现细节<br>建立积极的审查文化，以学习为主<br>审查者提供建设性意见，而非批评<br>审查者解释建议的原因和好处<br>鼓励讨论和交流<br>将审查作为团队学习的机会 |
| **代码审查工具** | 辅助工具举例：<br>Codacy：自动化代码质量检查工具<br>Code Climate：代码质量分析和监控平台<br>CodeGrip：代码审查和分析工具<br>CodeScene：代码演进分析和可视化工具<br>Reviewboard：代码审查平台<br>Smartbear：代码审查和协作工具<br>SonarQube：代码质量和安全分析平台<br>Veracode：应用安全测试平台 |

| 要点和思考方向 |
|----------------|
| 代码审查是保证代码质量的重要环节 |
| 通过代码审查共享知识，获得不同角度的思考 |
| 在代码合并前发现问题，避免bug进入主分支 |
| 让更多人了解系统实现细节，避免知识孤岛 |
| 建立积极的审查文化，将审查作为学习机会 |
| 使用自动化工具提高审查效率 |

| 类型 | AI Coding 指南 |
|------|----------------|
| **交互提示** | 说明需求；如「Code Review」「SonarQube」「检查清单」「审查规范」 |
| **方法** | 让 AI 设计审查清单、审查规范、工具配置 |
| **应用** | 审查清单、审查流程、自动化检查 |
| **提示词范例** | 「团队需 Code Review 流程，请设计审查检查清单（逻辑、安全、性能、可读性），以及 SonarQube/Codacy 与 Git 的集成配置要点」 |
