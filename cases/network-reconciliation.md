# 网络同步踩坑实录：MOBA 手游海外版"回弹"事故

> 一句话总结：客户端预测太傻、服务器校正太硬，200ms 延迟下角色「走一步弹回一步」，海外玩家差点把我们喷下架。

---

## 背景

项目是一款 MOBA 手游，技术栈是 Unity + C# 客户端、C++ 服务器，网络层走的是 UDP + KCP 可靠传输。国内上线半年，数据不错，玩家反馈也都正常，团队信心满满地开始推海外版。

结果东南亚上线第一周，客服后台就炸了。

> 「角色走路像在蹦迪，一卡一卡的」
> 「我往前走，走两步又回来了，这游戏没法玩」
> 「延迟显示是绿的，但画面根本对不上我的手」

我们一开始是懵的——国内测了那么久，从来没听说过这种问题。后来一查，国内玩家延迟普遍在 30ms 以内，而海外玩家走的节点延迟 200ms 起步，有些地区甚至到 350ms。

问题很明确了：**我们的网络同步方案，在高延迟下完全站不住脚。**

---

## 症状

具体表现非常一致，几乎所有海外玩家描述的体验都是同一个模式：

- 玩家摇杆往某个方向推，角色先走出去一段距离（动作流畅、动画正常）
- 大约 0.2 秒后，角色**突然弹回**到之前的某个位置，像被橡皮筋拽回去一样
- 然后角色继续走，过一会儿又弹一次
- 循环往复，就是「走一步弹回一步」

几个关键特征：

| 场景 | 表现 |
|------|------|
| 角色静止 | 位置完全正确，不弹 |
| 匀速直线移动 | 偶尔弹，频率较低 |
| 频繁变向（走位、拉扯） | 弹得最严重，几乎每一步都弹 |
| 延迟 < 50ms（国内） | 基本感觉不到 |
| 延迟 200ms+（海外） | 体验灾难级 |

用一个海外测试同学的话说：「我明明往前走想躲技能，结果画面先让我走过去了，然后突然把我拽回来，我直接吃了全套伤害。」

这就是经典的**客户端预测与服务器校正冲突**问题，但我们当时的实现有致命缺陷。

---

## 排查过程

### 第 1 步：怀疑网络丢包

第一反应肯定是丢包——KCP 虽然可靠，但如果底层丢包严重，重传会导致消息延迟，可能引发客户端长时间没收到校正包然后突然收到一堆。

加了丢包率监控后，数据显示：

- 海外节点平均丢包率 0.7%
- 峰值也不到 2%
- KCP 快速重传机制工作正常

**结论：丢包不是主因。**

### 第 2 步：怀疑服务器处理延迟

接下来怀疑是服务器那边处理不过来。毕竟海外玩家一多，服务器负载可能不一样。

在服务器移动处理逻辑里加了耗时打点：

```
[MoveHandle] playerId=1002345, cost=3.2ms
[MoveHandle] playerId=1002346, cost=2.8ms
[MoveHandle] playerId=1002347, cost=4.1ms
...
```

平均 3.5ms，99 分位也不到 5ms。服务器处理移动请求可以说毫无压力。

**结论：服务器不慢。**

### 第 3 步：在客户端加预测日志

这时候我开始怀疑客户端预测逻辑本身。在客户端每次收到服务器位置校正时，打印对比日志：

```
[Prediction] tick=3420, predicted=(102.3, 0, 56.7), server=(98.1, 0, 55.2), delta=4.5m
[Prediction] tick=3422, predicted=(103.8, 0, 56.9), server=(99.2, 0, 54.8), delta=5.1m
[Prediction] tick=3424, predicted=(105.1, 0, 57.2), server=(100.4, 0, 54.5), delta=5.4m
```

看到这个日志的瞬间我就明白了——**客户端预测的位置和服务器权威位置差了 4-5 米**，而且这个差距还在持续扩大。每次服务器校正包一来，客户端就把角色从 105.1 硬设到 100.4，视觉上就是「弹回去」。

### 第 4 步：审视预测逻辑

翻出客户端的预测代码，我直接倒吸一口凉气：

```csharp
// 原来的"预测"——说实话，这不叫预测，这叫瞎猜
void PredictPosition() {
    transform.position += currentVelocity * Time.deltaTime;
}
```

就这一行。它假设角色会以当前速度永远匀速直线运动下去。

但在 MOBA 里，玩家在做什么？频繁变向、急停、加速、放技能位移——这些操作在服务器上已经发生了（因为服务器提前收到了输入），但客户端在等待校正包的这 200ms 里，还在傻傻地按旧方向匀速外推。

**客户端预测的方向和服务器实际执行的方向完全不同，差距自然越拉越大。**

---

## 根因分析

把问题拆开看，其实是两个错误叠加在了一起：

### 错误一：客户端预测过于简陋

所谓「预测」只是 `velocity * dt` 的线性外推。这在一个玩家匀速走直线的情况下勉强能看，但 MOBA 玩家的移动特征是：

- 高频变向（走位、躲技能、拉扯对线）
- 短时加速/减速（技能效果、装备被动）
- 非连续位移（闪现类技能）

这些情况下，客户端根本不知道自己之前发出去的输入在服务器端已经被处理成了完全不同的运动状态。预测的基础假设已经错了，后面只能越错越离谱。

**正确的做法**：客户端应该维护一个输入缓冲队列，记录所有「已发送但未收到服务器确认」的输入。在收到服务器状态校正后，以服务器状态为起点，将未确认的输入全部重放一遍——这才是真正的「客户端预测」。

### 错误二：服务器校正采用硬设

```csharp
// 收到服务器校正包时，原来的处理
void OnServerCorrection(Vector3 serverPos) {
    transform.position = serverPos;  // 直接跳过去
}
```

没有任何过渡，没有任何平滑，就是一瞬间把角色从 A 点拽到 B 点。在 30ms 延迟下，A 和 B 的差距通常很小（可能不到 0.3m），人眼根本察觉不到。但在 200ms 延迟下，差距动辄 3-5 米，硬设就会产生肉眼可见的「瞬移回弹」。

**核心认知错误**：我们把 reconciliation 理解成了「用服务器位置替换客户端位置」。实际 reconciliation 的含义应该是「修正客户端的累积误差」，修正不等于替换——小误差可以容忍，大误差也应该平滑过渡，而不是硬设。

---

## 解决方案

针对上面两个根因，我们做了三件事：

### 1. 客户端预测：输入缓冲 + 重放

客户端不再做简单的线性外推，而是：

```
在每个 tick：
  1. 将玩家输入（方向、技能键等）加入未确认队列
  2. 发送输入到服务器
  3. 以当前权威状态为基础，重放未确认队列中的所有输入
  4. 得到预测位置
```

当收到服务器校正包时：

```
  1. 将服务器确认过的输入从未确认队列中移除
  2. 以服务器位置为起点
  3. 重放剩余未确认的输入
  4. 得到新的预测位置
```

这样，客户端的预测实际上是在**模拟服务器逻辑**——它用的输入序列和服务器一致，只是比服务器少了一轮 round-trip 的确认延迟。

### 2. 服务器校正：平滑插值

不再直接 `SetPosition`，改为逐帧插值：

```csharp
void OnServerCorrection(Vector3 serverPos) {
    // 平滑过渡：每帧向服务器位置靠拢 30%
    targetPosition = serverPos;
    isCorrecting = true;
}

void Update() {
    if (isCorrecting) {
        transform.position = Vector3.Lerp(
            transform.position, 
            targetPosition, 
            0.3f  // 插值系数，经过多轮测试的较优值
        );
        
        if (Vector3.Distance(transform.position, targetPosition) < 0.1f) {
            transform.position = targetPosition;
            isCorrecting = false;
        }
    }
}
```

`lerp` 系数 0.3 是我们反复调试的结果：太大了回弹感还在，太小了校正太慢、角色位置长时间不准。

### 3. 阈值判断：小误差不校正

加了一个简单的阈值逻辑：

```csharp
float correctionThreshold = 0.5f;  // 0.5 米以内懒得管

if (Vector3.Distance(currentPos, serverPos) < correctionThreshold) {
    return;  // 误差不大，忽略这次校正，避免微抖动
}
```

之前每次服务器校正到来都硬设，即使误差只有 0.1 米也会产生微小抖动，积累起来体感很差。加了阈值后，只有真正需要校正的偏差才会触发平滑过渡，大幅减少了画面微颤。

---

## 效果

我们在相同的测试环境下做了 A/B 对比（200ms 模拟延迟，30 秒自由移动）：

| 指标 | 改前 | 改后 |
|------|------|------|
| 回弹次数（30 秒内） | 15-18 次 | 2-3 次 |
| 最大回弹距离 | 5.4m | 1.2m |
| 玩家主观评分（1-10） | 2 分 | 7 分 |

改动上线到东南亚后，移动相关的客诉在一个星期内下降了约 85%。剩下来的反馈主要是「偶尔还会卡一下」，这个在 200ms+ 的极端延迟下确实无法完全消除，但已经从「没法玩」变成了「基本能接受」。

## 经验教训

### 1. 客户端预测必须模拟服务器逻辑，不是简单外推

「预测」的本质不是猜，而是**在客户端提前运行和服务器相同的计算**。你得有相同的输入序列、相同的状态机、相同的物理逻辑。简单外推在 FPS 里也许勉强能用（因为玩家确实倾向于持续移动），但在 MOBA 这种高频率变向的场景下毫无意义。

### 2. 校正要平滑，硬设必回弹

`SetPosition(serverPos)` 这种代码在局域网测试里从不会出问题，因为误差小到看不见。但这不是代码写得对，而是测试环境掩盖了问题。任何直接设置位置的校正方式，在延迟达到某个阈值后都会出现可感知的跳变。**平滑过渡不是优化，是必须。**

### 3. 阈值法简单有效——小误差忍了比修正更好

之前我们有一种「强迫症」心态：服务器位置是权威的，客户端必须完全一致。但实际上，画面渲染只需要「看起来对」——0.3m 的误差玩家根本看不出来，强行修正反而引入了不必要的抖动。**完美是好的敌人，尤其是在实时网络同步领域。**

### 4. 高延迟下先做体验测试，不要只看局域网

这条教训是用海外玩家的愤怒换来的。国内局域网环境太过理想，很多网络同步问题在这种条件下完全暴露不出来。**所有网络同步方案的验收标准，必须包含至少 200ms 模拟延迟的测试场景。** 可以用 Unity 的 Network Simulator 或者直接 `tc netem` 在 Linux 服务器上模拟。
